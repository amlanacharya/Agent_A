# Agentic Demand Forecasting Workspace

A governed, agentic demand forecasting platform that converts messy production data into replenishment recommendations, learns from each run, and stays auditable through approval gates and observable MLOps.

## Architecture

The platform separates concerns into a **governed core** and **bounded agentic extension paths**:

- **Governed core** — canonical data, feature generation, model training, promotion, replenishment policy, monitoring, and approvals. These layers are deterministic and contract-verified.
- **Agentic extension** — EDA adapters, custom feature engineering, and forecasting code are only generated by an agent when standard tools fail. Each layer is capped at three attempts. Failed attempts produce an exact failure report; successful ones get tests and a markdown card.
- **Durable knowledge** — all learning is stored in markdown artifacts (`LEARNINGS.md`, `DECISIONS.md`, `PROMOTION_DECISIONS.md`, etc.). Graph/memory systems act as indexes over these artifacts, not replacements.
- **Approvals + scheduling + ERP handoff** — the platform's own native layer (`approval_gateway.py`, `scheduler.py`, `erp_release.py`) handles human sign-off, scheduled jobs (data refresh, monitoring, drift), and the release-to-ERP payload.
- **Data Intelligence Cockpit** — a real-time observability UI that shows agent step, tool result, escalation status, attempt count, verifier gate, approval needed, confidence, and blockers.

## Canonical Demand Schema

All data paths converge on this schema before any feature or model work:

| Column | Type | Notes |
|---|---|---|
| `sku_id` | string | Product identifier |
| `location_id` | string | Store / DC identifier |
| `week_start` | date | ISO week Monday |
| `demand_qty` | float | Adjusted demand (not shipments) |
| `inventory_qty` | float | On-hand at week start |
| `stockout_flag` | bool | True if stockout occurred |
| `price` | float | Selling price |
| `promo_flag` | bool | Active promotion indicator |
| `lead_time` | int | Replenishment lead time in days |

## Hard Rules

- Agentic code generation is **escalation**, not default behavior.
- Each layer gets a maximum of **three** code-generation attempts.
- After three failures, the system stops and produces a structured failure report.
- The canonical schema is **immutable** even when adapters are custom.
- Markdown is the **durable source of truth**. Memory/graph systems index it; they do not replace it.
- Harnesses decide promotion; agents propose candidates and evidence.
- The platform's own approval workflow governs all approvals and downstream actions.

## Approval Gates (in-process)

The platform raises an `ApprovalRequest` on the `InProcessApprovalGateway` for any decision that needs a human (replenishment batches, custom-code permissions, schema sign-off, etc.). The cockpit UI lists pending requests from `gateway.list_pending(run_id)`, lets the user click Approve/Reject/Defer with a reason, and calls `gateway.acknowledge(request_id, decision, approver, reason)`. Every state change is written to `outputs/{run_id}/approvals.jsonl` for the audit trail. Approved batches flow into `release_erp_handoff` to produce the typed `ErpHandoffPayload` that the UI displays and the caller (a webhook, a cron job, anything that can call ERP) ships.

1. Data contract sign-off
2. Risky schema semantic mapping
3. Custom code generation permission
4. Unforecastable grain fallback
5. Official forecast publication
6. Replenishment recommendations
7. ERP / procurement handoff
